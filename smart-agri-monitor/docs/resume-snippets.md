# 智慧农业项目简历材料

## 简历项目描述

智慧农业设备监测与预警平台  
基于 Spring Boot 3 + MyBatis-Plus + MySQL + Redis + Vue 3 实现农业设备监测平台，支持 JWT 登录鉴权、地块与设备管理、模拟传感器数据采集、阈值规则预警、告警去重处理、ECharts 看板、历史查询和 EasyExcel 报表导出，并通过 Docker Compose 完成 MySQL、Redis、后端和前端的一键部署。

## 技术亮点

- 设计地块、设备、传感器数据、预警规则、告警记录等核心表结构，使用 MyBatis-Plus 完成分页查询和 CRUD 接口。
- 基于 JWT + Redis 实现登录鉴权和 token 服务端失效能力，避免纯 JWT 无法主动退出的问题。
- 通过 Spring 定时任务模拟设备上报温度、空气湿度、土壤湿度和光照数据，形成完整数据采集链路。
- 实现阈值规则匹配和告警去重，同一设备同一规则存在未处理告警时不重复生成，减少告警刷屏。
- 使用 Redis 缓存设备最新数据和看板短周期统计，降低高频看板刷新对 MySQL 的压力。
- 使用 EasyExcel 实现传感器数据和告警记录导出，配合 Swagger 提供接口文档，提升项目可演示性。
- 使用 Docker Compose 编排 MySQL、Redis、Spring Boot 后端和 Vue 前端，支持一条命令启动演示环境。

## 1 分钟自我介绍版

我做过一个智慧农业设备监测与预警平台，主要用 Spring Boot、MyBatis-Plus、MySQL、Redis 和 Vue 3 实现。项目里有地块、设备、传感器数据、预警规则和告警记录几个核心模块。后端定时模拟设备上报温度、湿度、土壤湿度和光照数据，数据入库后会匹配阈值规则，命中后生成告警，并做同设备同规则的未处理告警去重。前端可以查看看板、管理设备和规则、处理告警、查询历史曲线、导出 Excel。这个项目我重点练了接口设计、数据建模、JWT 鉴权、Redis 缓存、定时任务和 Docker Compose 部署。

## 3 分钟项目讲解版

这个项目的背景是农业设备监测。真实场景里，农田或温室会有很多传感器采集温度、湿度、土壤湿度和光照等数据。我的项目做了一个中等 MVP，不接真实硬件，而是用后端定时任务模拟设备数据上报，重点展示 Java 后端完整工程能力。

系统分为前端管理后台和后端 API。后端使用 Spring Boot 3，数据访问用 MyBatis-Plus，数据库是 MySQL，缓存用 Redis，登录鉴权用 JWT，接口文档用 Springdoc OpenAPI，报表导出用 EasyExcel。前端使用 Vue 3、Element Plus 和 ECharts，部署用 Docker Compose。

核心业务链路是：管理员登录后管理地块、设备和预警规则；后端定时生成传感器数据；每条数据入库后触发规则匹配；如果超过阈值，就生成告警记录；前端看板展示设备状态、今日数据量和告警趋势；用户可以在告警页处理告警，也可以在历史数据页查看曲线并导出报表。

我觉得这个项目最有价值的点有三个。第一是数据建模比较完整，包含主数据、时序数据、规则和告警处理记录。第二是告警规则不是简单插入，而是做了去重，避免同一设备同一规则持续刷屏。第三是 Redis 的使用比较贴近实际，包括 token 缓存、设备最新数据缓存和看板短缓存。

## 面试官可能追问

### 你这个项目和普通 CRUD 有什么区别？

普通 CRUD 只展示表单和接口，这个项目有持续数据采集、规则匹配、告警去重、缓存和报表导出，能讲出后端业务链路和工程取舍。

### 为什么用 Redis？

Redis 主要用于三处：登录 token 缓存，支持服务端主动失效；设备最新数据缓存，实时页面优先读取；看板统计短缓存，减少频繁刷新时的数据库压力。

### 告警去重怎么实现？

生成告警前查询同一设备、同一规则下是否已有 `pending` 或 `processing` 状态的告警。如果有，就不重复生成；如果告警已处理，后续再次命中规则可以生成新告警。

### 如果设备数据量很大怎么办？

先保证查询都带设备、指标和时间范围，并建立联合索引。再进一步可以按月分表，或者把传感器明细放到时序数据库，MySQL 只保留业务主数据和近期热数据。

### 项目还能怎么扩展？

可以接入 MQTT 或消息队列替代定时模拟数据；增加设备离线判断；把规则加载到 Redis；用 WebSocket 推送实时告警；补菜单权限和 CI/CD。
